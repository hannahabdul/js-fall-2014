Keeping Forked Git Repos in Sync

As the class has progressed, I've been committing changes to my repo, and you've been committing changes to your repo. But you might want to pull over the files I've created into your own repo, for instance, as I add more homework files. You're welcome to do this manually by creating new files inside your repo and pasting in the contents of files from mine, but it's usually easier to merge the changes from my repo to yours.

Normally, this is pretty easy, but it can get complicated if we've both changed the same file in ways that overlap. In that case, git doesn't know which is the "Right" version, mine or yours, and you have to manually tell it what to do. But there's a good chance you actually haven't done that, so here are directions to do this for this easy case.

First, you'll see that my forked repo is one commit behind the original.

Next you'll click the pull request button. Normally, a pull request is something you create when you have changes you want the owner of the original repo to add to theirs. But there's a way to flip that relationship around, so you'll be creating a pull request from the original (my repo) to yours.

To do this, click the text link that says "Try switching the base." This changes the base repo to be yours instead of mine.

You'll notice now your repo is on the left. Now you can click, "Create pull request."

Ok, now you're either lucky, or you're going to have to do things the hard way: If this screen says "Able to Merge", you're good. If it doesn't you can give up now. You have to pull these changes in manually and fix merge conflicts.

If you are able to merge, you can hit the "create pull request" button now.

Lastly, you'll merge this pull request into your repo by clicking the merge button.

Now, on the terminal, you should be able to `git pull` and you'll get all the changes I've made. Repeat this step periodically to keep up to date.
